## Data Model

Use a build-time Printful sync plus a local curation layer.

Generated Printful data:

```text
src/data/printful-products.generated.json
```

Generated by:

```text
npm run sync:printful-products
```

The sync script reads Printful sync products and variants from the Printful API, then normalizes:

- product ID and external ID
- product name
- preview image URLs
- prices
- colors
- variants
- SKUs
- catalog product names
- print file references

Local Fierce Highways curation:

```text
src/data/merch-curation.ts
```

This maps Printful sync product IDs to brand/editorial fields:

- slug
- display name
- kicker
- description
- product type
- Printful/Quick Store URL
- live or coming-soon status

Merged storefront data:

```text
src/data/merch.ts
```

This combines generated Printful data with the local curation layer and exposes a display-ready `merchProducts` array.

This gives us the useful part of automation without letting Printful own the brand language. Printful supplies the inventory facts; Fierce Highways supplies the meaning.

Move to an Astro content collection only if:

- Product pages become long-form editorial pages
- There are 10+ products
- Merch copy needs markdown bodies
- The store becomes a real brand surface instead of a lightweight front door

## Implementation Plan

### Phase 1: Static Store Page

1. Add `scripts/sync-printful-products.mjs`.
2. Add `npm run sync:printful-products`.
3. Generate `src/data/printful-products.generated.json` from Printful.
4. Add `src/data/merch-curation.ts` for local product voice, slugs, and Quick Store URLs.
5. Add `src/data/merch.ts` to merge generated Printful facts with curated Fierce Highways copy.
6. Create `src/pages/store.astro`.
7. Add `Store` to `src/components/Header.astro`.
8. Build responsive product cards using existing site utilities.
9. Use disabled-style CTAs for products with `status: 'coming-soon'` and live external links for known URLs.

### Phase 2: Polish

1. Add product color swatches if variant colors are worth showing.
2. Add a compact fulfillment note near the product grid.
3. Tune mobile spacing and image aspect ratios.
4. Add `store` metadata to `BaseHead` via page title/description.
5. Test with `npm run build`.
6. Run local preview and visually check desktop and mobile widths.

### Phase 3: Optional Product Detail Pages

Add `src/pages/store/[slug].astro` only if the landing page feels too cramped.

Use detail pages for:

- longer product copy
- mockup galleries
- variant explanations
- printing/fulfillment notes
- links to multiple Printful product variants

For now, avoid this. The first pass should sell the joke quickly and send people to checkout.

## Anti-Goals

Do not build:

- Shopify integration
- cart state
- checkout
- inventory logic
- discount codes
- email capture
- customer accounts
- overbuilt CMS machinery
- client-side Printful API calls

The point is a branded front door. The transaction still belongs to Printful.

## Acceptance Criteria

- `/store` exists and matches the Fierce Highways visual system.
- Header includes `Store`.
- Store page is responsive at mobile and desktop widths.
- Product cards include image, name, description, price/status, and Printful CTA.
- External links open safely with `target="_blank"` and `rel="noopener noreferrer"`.
- `npm run sync:printful-products` refreshes generated Printful product data.
- `npm run build` passes.
- No new runtime dependency is added.

## Future Decision Point

Revisit Shopify or another ecommerce platform only after one of these becomes true:

- Merch has enough demand to justify branded checkout.
- Product count grows beyond a small collection.
- Email capture, discounts, analytics, or abandoned cart flows become important.
- The Printful Quick Store experience becomes visibly conversion-limiting.

Until then, do not pay rent on an empire that has not yet conquered anything.
